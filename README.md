# FirmAE-modding

This repository is a firmware-emulation-only fork of [pr0v3rbs/FirmAE](https://github.com/pr0v3rbs/FirmAE).

The original FirmAE project includes firmware emulation, PostgreSQL-backed metadata management, vulnerability analysis, fuzzing, and firmware crawling. This fork keeps only the pieces needed to unpack firmware, build an emulation image, infer networking, start QEMU, and expose debug shells and reachable services. The goal is to run a single firmware image fully inside Docker without relying on a long-lived database or runtime bind mounts.

## Scope

- Firmware emulation and debugging only.
- No PostgreSQL dependency and no external database service.
- No automated vulnerability scanning, routersploit integration, web fuzzing, initialization crawler, or firmware crawler.
- Docker-first runtime. The Docker image contains the code and binaries needed for emulation.
- ASLR is disabled by default and can be enabled per run.

## Major Changes From Upstream

### Database Dependency Removed

Upstream FirmAE stores firmware, filesystem, brand, architecture, and analysis metadata in PostgreSQL. This fork removes that dependency chain:

- Removed `database/schema`.
- `install.sh` no longer installs or initializes PostgreSQL.
- `requirements.txt` no longer includes `psycopg2`.
- `run.sh`, `sources/extractor/extractor.py`, `scripts/tar2db.py`, `scripts/getArch.py`, `scripts/util.py`, and related scripts no longer connect to a database.
- Runtime state is kept in local `images/`, `scratch/`, and filesystem-generated intermediate files.

When running through Docker, no host-side PostgreSQL service is required and `PSQL_IP` does not need to be configured.

### Simplified Docker Entry Point

Upstream `docker-helper.py` supports brand names, batch modes, checks, and analysis modes. This fork reduces it to a single-firmware Docker emulation entry point:

```console
$ sudo ./docker-helper.py <firmware>
```

Optional ASLR controls:

```console
$ sudo ./docker-helper.py --aslr <firmware>
$ sudo ./docker-helper.py --no-aslr <firmware>
```

Behavior changes:

- No brand argument is required.
- Analyze/check helper modes are no longer exposed.
- Containers run `run.sh -d` so the debug shell is always enabled.
- The main emulation container stays alive after the firmware starts.
- The helper first tries to discover firmware TCP listeners through the debug shell.
- Only services that can actually be reached are published to the host.
- If debug-shell discovery is unavailable, the helper falls back to common service port mappings.

Common service ports include:

- Web: `80`, `443`, `8080`, `8443`
- SSH: `22`
- Telnet: `23`
- FirmAE debug netcat shell: `31337`
- FirmAE debug telnet shell: `31338`

If the matching host port is free, the helper tries to use the same host port as the firmware port. If the host port is already occupied, Docker assigns a random host port. The final access URLs and commands are printed after startup.

### ASLR Disabled By Default

Some real routers and small network appliances do not enable address space layout randomization. To better match those targets, this fork disables firmware-side ASLR by default:

```sh
FIRMAE_ASLR=${FIRMAE_ASLR:-false}
```

The selected value is written into `/firmadyne/aslr` inside the emulated root filesystem. `scripts/preInit.sh` and the debug startup logic generated by `scripts/makeNetwork.py` keep `/proc/sys/kernel/randomize_va_space` aligned with that setting.

Enable ASLR explicitly with:

```console
$ sudo ./docker-helper.py --aslr <firmware>
```

### Local Binary Inputs Instead Of Downloads

The binwalk source archive is installed from local `binaries/binwalk-2.3.4.tar.gz`. The binwalk extraction helpers `sasquatch`, `yaffshiv`, `jefferson`, and `ubi_reader` are still installed by binwalk's dependency flow from the network; they are required components, not optional extras.

### Dockerfile Optimized

`core/Dockerfile` has been adjusted for faster domestic builds and a smaller runtime image:

- apt and pip are switched to Tsinghua University mirrors before installing dependencies.
- binwalk is installed in a builder stage; the final image receives only the runtime `/usr/local` output.
- apt recommends and suggests are disabled.
- documentation, man pages, info pages, and similar non-runtime files are excluded.
- apt and pip caches are cleaned.
- Google Chrome, Chromedriver, Selenium, and analyzer-side dependencies are removed.
- headless Java packages are used.
- a dummy in-container `fuse-module` provider prevents `fusecram` from pulling unnecessary Linux kernel image/module packages.
- project files are copied through an intermediate stage so the final image preserves the repository layout without carrying the local binwalk source archive.
- the final image excludes removed analysis and crawler components.

The latest verified `fcore` image in the test environment is about `1.36GB`.

### Vulnerability Analysis, Fuzzing, And Crawling Removed

The following upstream components were removed:

- `analyses/`
- `analyses/fuzzer/`
- `analyses/initializer.py`
- `analyses/nmap.sh`
- `analyses/routersploit`
- `analyses/routersploit_patch`
- `sources/scraper/`
- `scripts/injectionChecker.sh`

Related behavior changes:

- `run.sh -a` and `--analyze` are no longer available.
- The image no longer installs `nmap`, `snmp`, `selenium`, `scrapy`, `requests`, `bs4`, `paramiko`, `pysnmp`, `pycryptodome`, or similar analysis-side dependencies.
- `scripts/makeImage.sh` no longer writes the `/bin/a` command-injection fuzzing helper into firmware images.
- This README does not document automated vulnerability analysis or CVE reproduction workflows.

This fork is no longer a vulnerability scanning framework. Use upstream FirmAE or restore the removed components if those workflows are required.

## Docker Usage

### 1. Prepare Local Binaries

Place all required files listed above into `binaries/`.

### 2. Build The Image

```console
$ sudo ./docker-init.sh
```

`docker-init.sh` first runs `download.sh` to validate local binaries, then builds the `fcore` image.

### 3. Start Firmware Emulation

```console
$ sudo ./docker-helper.py ./firmwares/example.bin
```

After a successful startup, the helper prints:

- main emulation container name
- port-forwarder container name
- inferred firmware IP
- service discovery method
- published services and access commands

Example output:

```text
[+] Firmware emulation is running.
[+] Container: firmae_<iid>_<firmware>_<timestamp>
[+] Port forwarder: firmae_<iid>_<firmware>_<timestamp>_ports
[+] Firmware IP: 192.168.0.1
[+] Published services:
    http      host 0.0.0.0:80 -> firmware 192.168.0.1:80      http://127.0.0.1:80/
    telnet    host 0.0.0.0:23 -> firmware 192.168.0.1:23      telnet 127.0.0.1 23
```

### 4. Access Firmware Services

Use the commands shown in the `Published services` section:

```console
$ curl http://127.0.0.1:<host_port>/
$ telnet 127.0.0.1 <host_port>
$ nc 127.0.0.1 <host_port>
```

Whether the host port matches the firmware port depends on host port availability. If the requested host port is occupied, Docker allocates a random host port. Always use the printed result.

### 5. Stop Emulation

Use the container names printed by the helper:

```console
$ sudo docker stop <container> <port_forwarder>
```

### 6. Clean The Environment

```console
$ sudo ./docker-clean.sh
```

Dry-run mode:

```console
$ sudo ./docker-clean.sh --dry-run
```

The cleanup script handles:

- FirmAE-related Docker containers
- QEMU processes
- mount points under `scratch/`
- loop, kpartx, and dmsetup devices
- tap and bridge network devices
- generated `scratch/` and `images/` files

## Local Non-Docker Entry Point

The original `run.sh` workflow is still available for lower-level debugging:

```console
$ sudo ./run.sh -c <brand> <firmware>
$ sudo ./run.sh -r <brand> <firmware>
$ sudo ./run.sh -d <brand> <firmware>
$ sudo ./run.sh -b <brand> <firmware>
```

Notes:

- `-c` checks inferred network and web reachability.
- `-r` starts emulation and keeps it running.
- `-d` starts the debug shell.
- `-b` is used for kernel boot debugging.
- `-a` and `--analyze` have been removed.

The Docker entry point does not require a brand argument. The local `run.sh` entry point keeps `<brand>` for compatibility with the original flow.

## Directory Layout

Kept source and input directories:

```text
binaries/          local kernels and helper binaries
core/              Docker build files and container helper binaries
firmwares/         optional firmware sample directory
scripts/           image creation, network inference, and QEMU startup scripts
sources/console    FirmAE console helper source
sources/extractor  firmware extractor
sources/libnvram   NVRAM hook library source
util/              cleanup and result helper scripts
```

Runtime-generated directories:

```text
images/            extracted filesystem tarballs and kernel extraction results
scratch/           per-firmware working directories, images, logs, and inferred metadata
```

`images/`, `scratch/`, and `firmwares/*` are excluded from the Docker image by default.

## Build And Verification Notes

The current version has been verified in the remote test environment:

- `sudo ./docker-init.sh` builds successfully.
- The `fcore` image builds successfully and is about `1.36GB`.
- `binwalk`, `sasquatch`, `yaffshiv`, `jefferson`, and `ubireader_extract_files` are available.
- `qemu-system-mips`, `qemu-system-arm`, and `qemu-system-x86_64` are available.
- `fusecram` and Java/Javac are available.
- The image does not contain `analyses/`, `sources/scraper/`, or `scripts/injectionChecker.sh`.
- The image does not contain `nmap`, `snmpwalk`, `snmpget`, `google-chrome-stable`, or `chromedriver`.
- The Python environment does not contain `requests`, `bs4`, `paramiko`, `pysnmp`, `Crypto`, `scrapy`, `selenium`, `flask`, `bluepy`, or `psycopg2`.
- `run.sh -a` fails and the usage output no longer mentions `--analyze`.

## Notes

- Docker commands usually require root privileges or membership in the docker group.
- Builds still require network access for apt, pip, and binwalk helper installation.
- Kernels and project helper binaries are never downloaded automatically; place them in `binaries/` first.
- A successful emulation startup does not guarantee that every firmware service is functional. Use the services printed by `docker-helper.py`.
- If port mapping does not look right, check the debug shell output and `scratch/<iid>/makeNetwork.log`.
